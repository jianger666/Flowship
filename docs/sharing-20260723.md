# Flowship：推进式 AI 工作台

> 部门技术分享 · 2026-07-23 · 陈禄江（前端）  
> **一句话：不是又一个 AI 编辑器，而是一种全新的 AI 工作方式——推进式工作台。**

---

## 0. 开场（约 3 分钟）

今天不讲「又一个包了 Cursor 的壳」，也不讲「AI 写代码有多神奇」。

想聊一件更具体的事：

**我们已经有了 Cursor / Codex，也有了团队自己的 harness——日常交付怎么真正跑得稳、推得动、传得开？**

两种大家都熟悉的用法：

| 方式 | 典型产品 | 优点 | 痛点 |
|---|---|---|---|
| 对话式 | Cursor / ChatGPT / Codex 聊天 | 灵活、随时问随时改 | 经验难沉淀；流程靠自觉；换人换会话就丢 |
| 全自动流水线 | 固定脚本 / 一键流水线 | 高效、路径清晰 | 路径写死；例外难插；人难介入中间决策 |

中间还有一块空地——**推进式**：

- 人扣扳机（点「推进」）
- AI 打一发（跑一个 action）
- 落一份文档（artifact）
- 人看一眼、点头，再扣下一发

这条路部门其实已经在走——领导搭的 **团队 harness（wk-harness）** 就是用 `wk:*` 指令把交付拆成可确认的步骤。  
Flowship 做的事，是给这条路装一个**驾驶舱**：指令变按钮，产物变面板，个人经验变可同步的团队资产。

> 今天三个关键词：**推进**、**action / skill**、**团队共享库**。

## 1. Flowship 是什么（约 5 分钟）

### 1.1 一句话

**Flowship = 站在 Cursor SDK 肩膀上的桌面 AI 工作台。**

它不替代 Cursor / Codex 的模型能力，也不替代团队 harness 的流程与知识；  
它管的是：**怎么把一次次 AI 调用，组织成可推进、可回看、可分发的工作流。**

### 1.2 工作单元：task + action

- **task**：一个需求的生命周期容器
- **action**：task 里的「一步」——每步产出一份 markdown **artifact**，落盘可追溯

内置常见 action：

| Action | 干什么 |
|---|---|
| `plan` | 读需求、摸代码、出技术方案 |
| `build` | 按方案改代码 |
| `review` | 换一个 agent 复审（防自己审自己） |
| `ship` | 提 GitLab MR / 提测相关 |
| `dev` | 联调相关 |

更重要的是后面的**自定义 action**。

### 1.3 交互范式：推进，不是聊到黑

```
新建 task → 选 action，点「推进」→ AI 产出 artifact → 人确认（HITL）→ 再推进下一步
```

- 步与步之间**必须有人确认**（HITL）
- 每步默认**新起 agent**（防 context 膨胀）
- review **强制换 agent**
- 产物进 **artifact 面板**，可回看、可当下一发上下文

一句话：**人决定何时开火，AI 打准这一发，文档把战果留下来。**

## 2. 核心概念三件套（约 8 分钟）

### 2.1 推进（Advance）

「推进」= 在 UI 上选一个 action，让 AI 执行一步。

| 手敲（Codex / 终端） | Flowship |
|---|---|
| 记得住指令名、参数、顺序 | 面板上点一下 |
| 产物散落在 doc repo / 本地 | 统一进 artifact 面板 |
| 确认靠自觉 / 口头 | HITL ack 是流程硬节点 |

推进不是「更炫的聊天」，而是**把交付节奏显性化**：这一步做完了吗？产物在哪？下一步谁确认？

### 2.2 自定义 action = 壳 + skill

1. **壳**：名称、placeholder、产出要求（这一步长什么样、交什么卷）
2. **skill**：一份 `SKILL.md`（Anthropic Agent Skills 标准格式）——agent 的剧本

> **action = 工作流里的「按钮」**  
> **skill = 按下后 agent 要遵守的「剧本」**

流程可配置，经验可封装：踩过的坑、某工程的改法，写进 skill 就能反复复用。

### 2.3 团队 skill 库（组共享库）

| 能力 | 含义 |
|---|---|
| 组共享库无感挂载 | 地址内置，app 自动 clone/pull；能力页出现「团队」分组 |
| 知识库镜像 | 有权限的人一键镜像知识库；没权限的同事也能用上规范与 skill |
| skill / action 上传下载 | 自己的 skill 一键上传；带 action 标记的，同事同步后推进面板自动出现同款 |
| wk 指令 action 化 | `wk:*` 变成可点的推进 action |

> **个人经验 → 团队资产；工作流可分发，而不是口口相传。**

同事侧几乎零配置：配好 GitLab token 即可。

## 3. 与团队 harness（wk-harness）的关系（约 8 分钟）

### 3.1 先尊重：团队 harness 已经把「物种」定了

领导搭的 wk-knowledgebase / wk-harness，是一套完整的 AI 交付操作系统：

- **知识层**：工程 module-map、全局索引、路由规则、manual-overrides
- **技能层**：工程级 `project-analyzer` + 全局 skill（含 `wk-harness`）
- **流程层**：`wk:*` 指令驱动交付
- **门禁层**：状态机 + 产物质检 + 写入拦截 + **owner 确认不可被 Agent 替代**
- **产物层**：写到 doc repo（如 `requirements/<REQ-ID>/biz-analysis.md`）

固定读取顺序：

```
module-map → 业务边界 → 实现约束 → 改动套路 → 才下钻源码
```

流程指令（示意）：

| 指令 | 阶段 |
|---|---|
| `wk:prd-review` | PRD 自检 |
| `wk:biz-analyze` | 业务分析 |
| `wk:biz-confirm` | owner 确认 |
| `wk:change-propose` / `wk:change-confirm` | 变更提议 / 确认 |
| `wk:repo-design` | 仓库方案 |
| `wk:repo-execute` | 执行写码 |
| `wk:repo-review` | 评审 |
| `wk:biz-verify` | 全局验收 |

状态由脚本推进（示意）：

```
DRAFT_BIZ_ANALYSIS → CONFIRMED_BIZ_PLAN → REPO_DESIGN_READY
  → REPO_IMPLEMENTED → REPO_DONE → INTEGRATION_READY / QA_READY
```

硬门禁：`doc-quality-gate` 产物质检、`wk-hook-guard` 写入拦截（未确认方案前禁止改代码）、owner 确认不可替代。

**方向对了，而且已经在干活。**

### 3.2 核心洞察：同一物种，不同形态

| 团队 harness（wk-harness） | Flowship |
|---|---|
| `wk:biz-analyze` / `wk:repo-design` | `plan` action（及对应自定义 action） |
| `wk:repo-execute` | `build` action |
| `wk:repo-review` | `review` action |
| 手敲 `wk:*` 指令 | 「推进」按钮 |
| doc repo 里的 md 产物 | artifact 面板可视化 |
| owner 确认不可替代 | HITL ack |
| python 门禁脚本 | 后置 deterministic check |

> **不是两套互相竞争的系统。**  
> 团队 harness 定义「该怎么走」；Flowship 提供「更好走」的驾驶舱 / GUI。

领导在 Codex 里手敲 `wk:*`，流程一样成立。  
Flowship 让同一套流程**看得见、点得动、传得开**——尤其方便不必先背整套指令与目录约定的同事。

### 3.3 定位口号

**Flowship = 团队 harness 的驾驶舱。**

harness 是引擎与交通规则；Flowship 是方向盘、仪表盘，以及「经验装进车载系统、全队可同步」的能力。

## 4. 今天已支持的能力（约 5 分钟）

按「已支持」讲；细节仍在打磨，形态已能 demo。

1. **组共享库无感挂载**  
   地址内置，自动 clone/pull；能力页「团队」分组；同事配好 GitLab token 即可。

2. **知识库镜像**  
   有权限的人一键镜像进共享库 `knowledge/`；没权限的同事也能用上全套规范 + skill。权限边界还在，使用面被打开。

3. **skill / action 上传下载闭环**  
   沉淀 skill（可挂 `.flowship-action.json`）→ 上传共享库 → 同事同步 → 推进面板出现同款 action。  
   **工作流本身变成可分发资产。**

4. **`wk:*` 指令 action 化**  
   9 个 `wk:*` 做成 9 个推进 action；轻壳 skill 指向镜像里的 `wk-harness`；手敲指令 → 可视化点按钮。

## 5. Live Demo 流程脚本（约 10~12 分钟）

> 边点边说。只证明四件事：同步进来 → action 出现 → 推进一步看产物 → 上传形成分发闭环。  
> 不追求现场做完整需求。

### 5.1 同步共享库

1. 打开 Flowship → 能力页，确认「团队」分组可见  
2. 展示已同步 / 触发 pull  

**口播**：同事不用记共享库地址，也不用手 clone；token 配好，团队能力自己出现。

- [ ] 「团队」分组可见  
- [ ] 能看到 wk / 镜像相关 skill（名称以现场为准）

### 5.2 推进面板出现团队 action

1. 打开（或新建）task → 打开「推进」  
2. 指出由共享库同步下来的 `wk:*` action  

**口播**：这些不是本机手配的按钮，是共享库里的 skill + action 标记同步进来的——harness 里敲的指令，在这里变成可点的一步。

- [ ] 推进列表可见团队 action  

### 5.3 推进一次 `wk:biz-analyze`（或等价项）

1. 选中 → 点「推进」→ 等交卷 → 演示 HITL 确认  

**口播**：人扣扳机 → AI 打一发 → 落文档 → 人确认。和 harness「owner 确认不可替代」同一原则，确认从口头变成流程节点。

- [ ] action 运行 → 完成 / 待确认  

**备用**：模型慢或权限不足时——先展示「推进被触发、确认位在」，产物用预先准备的 artifact 演示面板。

### 5.4 artifact 面板看产物

1. 打开刚产出（或预留）的 markdown  
2. 指出历史 action 可回看、可给下一发当上下文  

**口播**：harness 里 md 写在 doc repo；Flowship 把同一类产物可视化、按 action 串起来。

- [ ] artifact 可读，有「第 N 步」的历史感  

### 5.5 上传自己的 skill，演示分发闭环

1. 能力页选一个预准备的小 skill → 一键上传  
2. 说明：同事下次同步后，若带 action 标记，推进面板会出现同款  

**口播**：不是再写一份 README 让大家自觉遵守，而是经验封装成 skill、工作流封装成 action，同步即分发。

- [ ] 上传成功；能说清同事 pull 后会发生什么  

### 5.6 收束（30 秒）

> 同步 → 按钮出现 → 推进一步 → 看产物 → 上传分发。  
> 引擎还是团队 harness，车开起来更容易。

## 6. 背后的 Harness 机制（约 5 分钟）

UI 好看不够；让 AI 别乱跑的是缰绳。与团队 harness 门禁哲学一致：

| 机制 | 做什么 |
|---|---|
| HITL | action 跑完待人确认，再进下一步 |
| Artifact | 每步 markdown 落盘；步间靠产物通信，不靠无限变长的聊天 |
| Deterministic check | 必备段 / 长度门槛；review 工作区指纹防偷改；ship 验 MR URL 等 |
| 换 agent 复审 | 默认每 action 新 agent；review 强制换人 |

团队 harness 用 python 做状态机 / 质检 / 写入拦截；Flowship 在 action 边界做后置 check——**同一种思想：LLM 非确定性，用确定性工具压住。**

其它集成（略）：飞书 story、GitLab 提 MR、MCP、多仓 worktree 并行。细节会后可看。

## 8. FAQ（约 5 分钟，可穿插）

### Q1. 和直接用 Cursor 什么区别？

| | Cursor（对话式） | Flowship（推进式） |
|---|---|---|
| 交互 | 持续聊天 | 按 action 推进 |
| 产物 | 散落在对话 / 本地改动 | 每步 artifact 落盘可视化 |
| 流程 | 靠个人习惯 | 可配置 action + HITL |
| 团队资产 | 难同步 | skill / action 可进共享库 |
| 定位 | 强力 IDE / Agent | 工作台 / 驾驶舱 |

Cursor 仍是引擎之一。差别是：**在聊天里碰运气，还是在工作流里推进。**

### Q2. 和 wk-harness 什么区别？会不会重复建设？

**不会按「再造一套流程」的方式重复建设。**

- wk-harness：流程本体、知识、门禁、状态机、指令协议  
- Flowship：UI、推进编排、artifact 可视化、团队 skill/action 分发  

同一物种，不同形态（见第 3 章对应表）。

> **harness 定义正确的路；Flowship 让更多人能按这条路开。**

### Q3. 是不是只适合前端？

不是。推进式 + skill 是通用壳；前端只是切入最熟的场景。后端 / 全栈可挂自己的 skill 与 action。

### Q4. 自定义 action 会不会很重？

壳可以很轻。重的是 skill 里的领域知识——本来就该沉淀，不该每次靠聊天现场编。

### Q5. 没 wk-knowledgebase 权限怎么办？

走**知识库镜像**：有权限的人镜像进共享库后，同事通过 Flowship 同步即可用（具体策略以组内约定为准）。

### Q5.5 镜像不含源码（codes/），skill 会不会变弱？

不会。两种模式方向相反、能力等价：

- 团队 harness 原版：人坐在知识库仓里，源码拉进 `codes/`——**把源码搬到知识旁边**
- Flowship：任务本来就绑真实业务仓，agent 脚下就是真源码，知识库镜像通过 kbRoot 另外供给——**把知识搬到源码旁边**

「先读档案 → 再下钻源码」的流程两边都成立；档案里 `codes/<group>/<repo>/` 开头的证据路径，在 Flowship 里映射到当前任务仓根目录即可。唯一差异：跨工程源码级下钻（想看不在本地的依赖工程源码）时，靠档案回答；真需要时把那个仓加进任务的多仓列表即可（按需拉指定仓，不用全量 clone）。

### Q6. AI 会不会跳过确认直接改代码？

HITL 是节点；review 有指纹类校验；团队 harness 还有写入拦截与状态机。**不相信自觉，相信门禁。** 两边哲学一致。

### Q7. 出了问题怎么回看？

看该 task 的 action 历史 + 对应 artifact；必要时从某步产物再推进。比翻一长串聊天更可操作。

### Q8. 我现在最该做的第一步？

三选一：装上跟一个小需求推进两步；把已有 checklist 改成 skill 上传；直接体验同步后的 `wk:*` action，对照 Codex 手敲习惯。

## 9. 收束（约 1 分钟）

> **Flowship 不是又一个 AI 编辑器，而是一种推进式 AI 工作方式。**

对部门更准确的说法：

> **它是团队 harness 的驾驶舱——让同一套流程更好用、更好推广，让个人经验变成团队资产。**
