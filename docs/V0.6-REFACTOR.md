## 1. 背景：为什么要重构

### 1.1 用户真实工作流（用户原话+复述）

```
飞书项目接到需求
  ↓
fe-ai-flow 新建 task、贴飞书链接、可能续补 PRD / 后端方案 / 测试用例
  ↓
plan → build → review → 提 MR v1
  ↓
任意阶段都可能回头改（产品变更 / review 问题 / 测试报 bug）
  ↓
改完、再提 MR v2 / v3 / ...
  ↓
最终完成（测试过 + 合入 main）
```

外加：「如果有测试用例 / 单元测试、AI 可以自测」。

### 1.2 当前 V0.5 phase chain 跟这个流程的 4 个错配

1. **单 task = 多个 MR**：现在 phase chain 隐含「单 task = 单次走 plan→build→review、结束」。用户实际是「单 task 容纳一个需求的整个生命周期、可反复提 MR」。
2. **MR 后回流机制缺失**：测试报 bug 后用户怎么操作？现在只能 fork from build（会重跑 review）或者切 chat 模式（另一套架构、不联动）。都不顺。
3. **测试用例 / 自测在 ROADMAP 里被标「不做」**——但飞书项目里的「测试用例」是人写的文字描述、AI 可以**手测**（起服务 / curl / 看页面表现）、跟 ROADMAP 想排除的「跑 `npm test` 单测」是两回事。
4. **review 跟 plan / 飞书原文的偏差只能单方向校验**——现在 review 发现 plan 错了能 edit 01-plan.md（V0.5.12 破例）、但 build 发现 plan 错了只能走推进 + revise、不顺畅。

### 1.3 用户拍板的方向

「自由度高一点」——任何 action 都能随时触发、不被 phase 顺序绑住。

同时承诺用 harness 思想守住质量（详见第 6 节 6 个门槛）。

## 2. 当前 V0.5 架构的核心问题（明确列、避免重构时漏）

### 2.1 phase 状态机太刚性

`Task.phases: { plan: {...}, build: {...}, review: {...} }`——固定 3 个 phase、各自有状态（pending / running / awaiting_ack / completed / error）。

问题：
- 「现在 build 第二次跑」表达不出来——只能覆盖之前的 `build.artifact`、丢历史
- 「现在跑测试」表达不出来——没有这个 phase
- 「现在提 MR」表达不出来——没有这个 phase（review 里塞了 commit msg 草稿、但不实际提）

### 2.2 单 SDK Run 跟 phase chain 绑死

`plan-runner.ts` 内部知道 phase 顺序（plan → build → review）、用 prompt 模板告诉 agent「按序跑完三个 phase」、最后一个 phase approve = Run 退出。

问题：
- 想「只跑 build action 不跑 review」做不到——Run 起来就走完整 chain
- 想「Run 永生、用户随时指令下一 action」做不到

### 2.3 chat 模式跟 plan workflow 是两套架构

- `plan-runner.ts` vs `chat-runner.ts`
- 任务详情页 plan 模式跟 chat 模式 UI 不同
- 同一个 task 不能切换模式

但用户视角「自由 action」跟 chat 模式本质相似（用户每次输入指令、agent 接着跑）、只是缺结构化 artifact。所以新架构会让两者合并。

### 2.4 推进 dialog 三选一不够表达

`resume / fork / restart`——表达「续接 / 切下个 agent / 重做」、但表达不了「换个 action 类型」。

V0.6 推进 dialog 要改成「**选下个 action + 写指令**」（详见第 8 节）。

### 2.5 phase 顺序是 V0.5 唯一的隐性 harness

phase 顺序本身约束了 agent 行为——你不能跑 review 因为 build 还没跑、就强迫 agent 按合理顺序走。

V0.6 拆掉顺序约束、要用**显性 harness** 补回来（第 6 节 6 个门槛）。

## 3. 新架构总览：task 容器 + action 历史

### 3.1 核心抽象

```
task（一个需求的容器、从飞书 story 进来到合入 main / abandon）
  │
  ├─ 仓库状态: 开发中 / 待测 / 有 bug / 已合入 / abandon
  ├─ MR 列表: [{ url, version, status, createdAt }, ...]
  ├─ action history（按时间正序、N 是累计序号）:
  │     1. plan      → actions/1-plan.md
  │     2. build     → actions/2-build.md
  │     3. review    → actions/3-review.md
  │     4. ship      → actions/4-ship.md   (MR v1)
  │     5. build     → actions/5-build.md  (测试报 bug、回 build)
  │     6. ship      → actions/6-ship.md   (MR v2)
  │     7. test      → actions/7-test.md   (AI 手测)
  │     8. build     → actions/8-build.md  (手测发现新 bug)
  │     9. ship      → actions/9-ship.md   (MR v3)
  │     ...
  │    10. learn     → actions/10-learn.md (task 标记完成后跑一次)
  │
  └─ contextDocs / mcp / role / repoPaths 等（保留 V0.5 字段、语义不变）
```

### 3.2 6 + 1 种 action

| action id | 中文 label | 含义 |
|---|---|---|
| `plan` | 出方案 | 读 contextDocs + 扫仓库 + 出 / 改技术方案 |
| `build` | 改代码 | 按最新 plan 写代码 + 跑 typecheck/lint |
| `review` | 代码复核 | 跑 git diff × plan、出 4 类差异报告 |
| `ship` | 提 MR | git commit + push + gh pr create + 飞书回写 |
| `test` | AI 手测 | 读飞书测试用例、起服务 / curl / 看页面、出测试报告 |
| `learn` | 沉淀 | task 标完成时、扫整个 action history、propose 仓库约定 → HITL 落 AGENTS.md / .cursor/rules / `prompts/_super.md` |
| `chat` | 自由聊 | 不结构化对话、agent 答疑 / 探索（吸收原 chat 模式）|

**注**：action 不是「角色」、是「动作类型」。同一个 agent 可以跑多种 action（不像 BMAD 那种 multi-agent 路线）。

### 3.3 单 SDK Run 永生

- 整个 task 只有一个 SDK Run（保留 V0.5 单 Run 决策、省 Cursor 计费）
- Run 不再因「最后一个 phase approve」退出、而是因「task 标记完成 / abandon」退出
- 每次 action 完成、agent 调 `wait_for_user`、用户在 UI 选下个 action + 指令、agent 拿到指令接着跑
- 保活机制完全沿用 V0.3.5 shell + curl long-poll（**不动**）

### 3.4 跟 V0.5 兼容度

| 项 | V0.5 | V0.6 | 兼容性 |
|---|---|---|---|
| 单 SDK Run | ✓ | ✓ | 沿用 |
| HITL 阻塞 | ✓ | ✓ | 沿用 |
| 保活 shell+curl | ✓ | ✓ | 沿用 |
| phase 概念 | ✓ | ✗（→ action）| **大改** |
| artifact 命名 | `01-plan.md` 等 | `N-plan.md` 等 | **大改** |
| 推进 dialog 三选一 | ✓ | ✗（→ 选 action）| **大改** |
| chat-runner | ✓ | ✗（→ action=chat）| **删 / 合并** |

## 4. task schema 改造

### 4.1 旧 schema（V0.5）

```typescript
interface Task {
  id: string;
  title: string;
  status: "pending" | "running" | "awaiting_user" | "completed" | "error" | "paused";
  currentPhase: "plan" | "build" | "review" | null;
  phases: {
    plan: PhaseState;
    build: PhaseState;
    review: PhaseState;
  };
  // ... 其他字段
}

interface PhaseState {
  status: "pending" | "running" | "awaiting_ack" | "completed" | "error";
  artifactPath: string | null;
  startedAt: string | null;
  endedAt: string | null;
}
```

### 4.2 新 schema（V0.6）

```typescript
interface Task {
  id: string;
  title: string;

  // 任务级状态机（新）：跟 MR 生命周期对齐
  repoStatus:
    | "developing"     // 默认、还在 plan/build/review/ship 循环
    | "awaiting_test"  // ship 完、等测试反馈
    | "has_bug"        // 测试 / review 反馈有 bug、回 build
    | "merged"         // MR 合入 main、可以触发 learn
    | "abandoned";     // 不做了

  // 运行时状态（保留、独立于 repoStatus）：
  runStatus: "idle" | "running" | "awaiting_user" | "error";
  currentActionId: string | null;  // 正在跑的 action 的 id（如果有）

  // action 历史（新核心）
  actions: ActionRecord[];

  // MR 列表（新）
  mrs: MRRecord[];

  // 保留字段（语义不变）
  contextDocs: ContextDoc[];
  role: TaskRole;
  repoPaths: string[];
  agentModel?: string;
  lastAgentId?: string;
  disabledMcpServers?: string[];
  // ... 其他 V0.5 已有字段
}

interface ActionRecord {
  id: string;               // 任务内唯一 id（如 act_1, act_2）
  n: number;                // 累计序号、对应文件名 `N-<type>.md`
  type: "plan" | "build" | "review" | "ship" | "test" | "learn" | "chat";
  status: "running" | "awaiting_ack" | "completed" | "error" | "cancelled";
  userInstruction: string;  // 用户在推进 dialog textarea 写的指令
  artifactPath: string | null;  // 相对路径 `actions/N-<type>.md`、chat action 可空
  startedAt: string;
  endedAt: string | null;
  // 后置检查结果（第 6 节门槛 2）
  postCheck?: {
    passed: boolean;
    details: string;  // typecheck 输出 / git diff hash / ...
  };
  // 副作用记录（ship action 用、未来 test action 也可能用）
  sideEffects?: {
    mrUrl?: string;
    feishuCommentId?: string;
    // ...
  };
}

interface MRRecord {
  version: number;          // v1, v2, v3
  url: string;              // GitHub PR URL（或 GitLab MR）
  branch: string;
  status: "open" | "closed" | "merged";
  createdAt: string;
  createdByActionId: string; // 哪个 ship action 创建的
  // 可选 polling 拿来的状态
  lastChecked?: string;
}
```

### 4.3 文件系统改造

**旧**：

```
data/tasks/<id>/
  meta.json
  events.log
  artifacts/
    01-plan.md
    02-build.md
    03-review.md
    .revisions/...     # 修订快照
```

**新**：

```
data/tasks/<id>/
  meta.json               # schema 改造
  events.log              # 不动
  actions/
    1-plan.md
    2-build.md
    3-review.md
    4-ship.md
    5-build.md
    6-ship.md
    ...
    .revisions/           # 修订快照、key 改为 actionId（不再是 phaseId）
      act_5/...
```

### 4.4 迁移策略（用户拍板：老 task 只读 / 归档）

- V0.6 启动时、老 task（schema 没 `actions` 字段）：UI 上标「Legacy V0.5 task」、只读模式、artifact 能看、不能继续推进
- **不写 migration 脚本**——按 project-context.mdc「开发期不写向后兼容代码、改 schema 直接删旧」
- 用户想继续做老 task 的内容：新建一个 V0.6 task、把老 task 的飞书链接 + contextDocs 手动贴过去（极少数场景、不优化）

## 5. 6 种 action 详细规格

每个 action 一个小节。统一字段：

- **触发条件**：在哪些 task 状态 / 历史下可选
- **agent 在 Run 内的行为**：read 哪些 artifact、调哪些 MCP、写哪个 artifact
- **artifact 骨架**：md 文件大致结构（详细骨架挪到 `prompts/action-<type>.md`）
- **后置检查（deterministic）**：跑完怎么判过没过
- **常见反例 / 雷区**

### 5.1 `plan` action

**触发条件**：
- task 创建后第一个 action（强烈建议、不强制）
- 用户在 build / review / ship 之后发现「方案需要调整」时可手动触发

**agent 行为**：
1. read 当前 task 的 contextDocs（飞书 story / PRD / 接口文档等）
2. 必要时 read 仓库代码（V0.3.4 起 plan 就该读仓库）
3. 跟用户 ask_user 澄清不确定项（不积压）
4. 写 `actions/N-plan.md`、第 N 个 plan 时引用上一个 plan + 标 diff（不重写）

**artifact 骨架**：复用 V0.5 `phase-1-plan.md` 骨架（§1 需求理解 / §2 业务规则 / §3 涉及接口 / §4 关键技术决策 / §5 task 拆分 / §6 待澄清）。

**后置检查**：artifact 文件存在 + 内容长度 >= 100（V0.5.6.5 黑名单 grep 已删、V0.6.0.1 拍板）。

**常见反例**：
- 不 ask_user 强行 propose 字段名（V0.5.6.3 已防）
- 第二次 plan 不读上一个 plan、写一份独立的（应该是 diff / 增量）

### 5.2 `build` action

**触发条件**：
- 至少有 1 个 `plan` action（**硬门槛**）

**agent 行为**：
1. read 最新 plan + 上一个 build（如果有）
2. 按 plan §5 task 拆分写代码
3. 跑 typecheck + lint（不跑 build 命令、V0.5.15 已拍板）
4. 写 `actions/N-build.md`、含 task 完成情况 + 校验结果

**artifact 骨架**：复用 V0.5 `phase-2-build.md` 骨架（V0.5.15 改完的版本、无独立验收对照表 / 文件清单 / 给用户的交接段、有 task 子条的 5 字段）。

**后置检查**（deterministic、第 6 节门槛 2）：
- `pnpm typecheck` exit 0
- `pnpm lint` exit 0
- git status 真实有改动（防 agent 假装写了代码）

**常见反例**：
- 改 plan 范围外的文件
- 跳过 typecheck / lint
- 不读上一个 build、把别人改的覆盖了

### 5.3 `review` action

**触发条件**：
- 至少有 1 个 `plan` + 1 个 `build` action（**硬门槛**）

**agent 行为**：
1. read 最新 plan + 最新 build
2. 跑 git diff（main..HEAD 或 origin/main..HEAD）
3. 出 4 类差异报告（保留 V0.5.12 review 闭环：实现偏差 / 未完成 task / 跳书未覆盖 / 跳代码未覆盖）
4. 不能 ask_user 决策的 → 落 `actions/N-review.md` 的「§ 用户决策」段、等用户 ack
5. 能 ask_user 拍板的 → 调 ask_user 一次性问完
6. 答完按 V0.5.12 规则落地（edit 最新 plan 的对应段、补 strikethrough）

**artifact 骨架**：复用 V0.5 `phase-3-review.md` 骨架（V0.5.15 拿掉 build 校验描述后的版本）。

**后置检查**：
- review artifact 的「git diff hash」字段跟实际 `git rev-parse HEAD` 一致（防 agent 编造 diff）
- 「§ 4 类差异」段非空（哪怕只是「无偏差」也要明确写）

**常见反例**：
- 不跑 git diff 凭印象写
- 把测试反馈 / 飞书评论塞 review（review 是「代码层差异」、不是「业务反馈」）

### 5.4 `ship` action（新增、V0.5 没有）

**触发条件**：
- 至少有 1 个 `build` action（**硬门槛**）
- `git status` 当前分支有改动 / 至少有 1 个未推送 commit（**硬门槛**）

**agent 行为**：
1. read 最新 build + review
2. 生成 commit message（基于最新 build artifact 的「task 完成情况」段）
3. 生成 PR title + body（基于飞书 story + build + review）
4. 调 `shell` 跑：
   - `git checkout -b <branch>`（如果还在 main）/ `git add` / `git commit`
   - `git push -u origin <branch>`
   - `gh pr create --title ... --body ...`（或 GitLab CLI、待用户拍板）
5. 拿到 MR URL 后、调飞书 MCP（user-feishu-project-mcp）把 URL 评论回飞书项目工作项
6. 写 `actions/N-ship.md`：commit hash / MR URL / branch / 飞书回写状态

**artifact 骨架**（新）：

```markdown
---
action: ship
n: <N>
mr_version: v<N_ship>
created_at: ...
---

## 1. 本次 ship 摘要
- 来源 build: actions/<X>-build.md
- 提的 MR：v<N>、url=...

## 2. commit
- branch: <feature/xxx>
- commit hash: <abc1234>
- message:
  ```
  <commit message 全文>
  ```

## 3. PR / MR
- title: <...>
- body:
  ```
  <PR body 全文>
  ```
- url: <https://...>

## 4. 飞书回写
- 工作项：<...>
- 评论 id: <...>
- 状态：✅ / ❌（失败 + 原因）

## 5. 后续等待
- 等测试反馈、task 状态自动转为 awaiting_test
```

**后置检查**：
- `git push` exit 0
- `gh pr create` exit 0、拿到的 URL 非空
- 飞书评论调用 exit 0（失败也不阻塞 ship、artifact 标 ❌、用户后续补）

**常见反例**：
- 强推到 main（**硬门槛禁**）
- commit message 跟 build artifact 不一致
- 不更新 `Task.mrs` 列表（runner 侧必须写入）

**待用户拍板**（第 13 节）：
- gh CLI vs GitLab CLI vs 自建（用户公司用什么）
- branch 命名规则
- 默认目标分支（main / develop）
- 飞书工作项评论的 MCP 工具名 + 入参

### 5.5 `test` action（新增、V0.5 没有）

**触发条件**：
- 至少有 1 个 `build` action（**硬门槛**）
- contextDocs 里有标记为「测试用例」的 doc（**软门槛**、没有时 warning「无测试用例、AI 自由探索」）

**agent 行为**：
1. read 最新 build + 飞书测试用例 doc
2. 起服务（如果还没起）—— `shell` 跑 `pnpm dev` 或类似（具体命令在 plan 里拍板）
3. 按测试 case 逐条手测：
   - case 描述「点 X 按钮、应该弹 Y」→ agent 调 `shell` curl / 调 puppeteer MCP 模拟、看返回 / DOM、判断 pass/fail
   - case 没法自动化（需要肉眼判断）→ 标 SKIP + 原因
4. 写 `actions/N-test.md`：case 列表 + pass/fail/skip 状态

**artifact 骨架**（新）：

```markdown
---
action: test
n: <N>
source_build: actions/<X>-build.md
---

## 1. 摘要
- 总 case 数: <N>
- pass: <X> / fail: <Y> / skip: <Z>

## 2. case 详情

### case 1: <case 标题>
- 来源：飞书测试用例 §<X>
- 预期：<...>
- 操作：<agent 用什么工具做的>
- 实际：<返回 / 截图 / 错误>
- 结论：✅ pass / ❌ fail / ⏭ skip
- 失败原因（如 fail）：<...>

## 3. 总结建议
- 必须修的 case：<list>
- 建议优化的：<list>
- 跳过的：<list>
```

**后置检查**：
- pass 率 ≥ plan 里定的阈值（默认 100%、用户在 plan §4 关键技术决策里可改）
- 不通过 → artifact 标 ❌ → 用户下一步推 build 修

**常见反例**：
- 编造 case 结果（必须配合 grep evidence、artifact 里有 shell 命令实际输出）
- 跳过太多 case（skip 率 > 50% → 后置检查标 ❌、提示用户「测试不充分」）

**待用户拍板**：
- 「测试用例」在飞书项目里的具体位置（子工作项 / 独立文档 / wiki）
- 起服务命令（plan 里 explicit 列）
- 端口冲突 / 数据准备怎么办（V0.6.2 上线时再讨论）

### 5.6 `learn` action（新增、V0.5 没有）

**触发条件**：
- `repoStatus = "merged"`（用户标记 task 完成 / 合入 main）（**硬门槛**）
- 整个 task 里**只能跑一次**（**硬门槛**）

**agent 行为**：
1. read 全部 action history + 当前的 `<repoPath>/AGENTS.md` + `<repoPath>/.cursor/rules/*.mdc`（避免重复）
2. 反思：哪些约定值得跨任务沉淀？
3. 写 `actions/N-learn.md`：propose 列表、每条带 trigger / rule / rationale / evidence / 落点建议
4. wait_for_user

**artifact 骨架**（新）：

```markdown
---
action: learn
n: <N>
covered_actions: [1, 2, 3, ..., <last>]
---

## 值得沉淀的约定（agent propose、HITL 落库）

### 1. <短标题>
- **trigger**：<场景>
- **rule**：<具体怎么做>
- **rationale**：<为什么>
- **evidence**：本次 action <X> 踩到、详见 actions/<X>-<type>.md §<Y>
- **落点建议**：`<repoPath>/AGENTS.md` / `<repoPath>/.cursor/rules/<topic>.mdc` / `prompts/_super.md`

### 2. ...

## 本次无值得沉淀的（如果有的话写这条、否则不写）
```

**HITL 落库**（V0.6.0 第一版可只 propose 不落库 / 用户手动复制；V0.6.3 加 dialog 勾选 + 自动写文件）：
- 用户在 learn 的 ack dialog 里看 propose 列表、勾选哪些进库、选落点
- approve 时 fe-ai-flow 自动写到目标文件（append、不去重、第一版简单）
- 落点 3 选 1：
  - `<repoPath>/AGENTS.md`（2026 emerging 跨工具标准、Cursor / Claude Code / Windsurf 都识别）
  - `<repoPath>/.cursor/rules/<topic>.mdc`（Cursor 单工具）
  - `prompts/_super.md`（fe-ai-flow harness 自我增强）

**后置检查**：
- propose 段不能全是「本次无」（防搪塞、强制让 agent 写至少 1 条；或者明确写「本次无、理由：...」）
- evidence 必须指向某个 action 的 artifact（防编造）

**常见反例**：
- propose 跟 `<repoPath>/AGENTS.md` 已有内容重复
- propose 没 evidence（凭印象、编造）
- propose 是临时 workaround（不该沉淀）

### 5.7 `chat` action（吸收原 chat 模式）

**触发条件**：
- 永远可触发

**agent 行为**：
- 用户在 textarea 写问题 / 指令
- agent 自由回复、可 read 任意 artifact / 跑 shell / 调 MCP
- 不强制写 artifact（artifact 可空、用户视角是「聊一聊」）

**artifact**：
- 可选：如果 chat 涉及探索性结论（如「我看了下、那个接口的字段是 X」）、agent 可选择写 `actions/N-chat.md` 记录关键结论
- 默认：不写、只在事件流里显示对话

**后置检查**：
- 无（chat 是探索性、不强制 deterministic 检查）

**常见反例**：
- chat 里偷偷改代码（chat ≠ build、改代码必须显式跑 build action）
- chat 写超长 artifact（chat 的 artifact 应该是简短结论、长的内容应该走 build / plan / review）

## 6. 6 个 harness 门槛

> 核心思路：phase 顺序拆掉后、用 deterministic 检查 + 软引导 + prompt 约束补回保证能力。

### 6.1 优先级总览

| 门槛 | 优先级 | 类型 | 落点 |
|---|---|---|---|
| 1. action 前置准入条件 | **P0、必做** | 硬 / 软 | runner + UI dialog |
| 2. action 后置 deterministic 检查 | **P0、必做** | 硬 | runner |
| 3. 默认 action = 推荐项 | P1 | 软 | UI dialog |
| 4. action 级 anti-patterns prompt | **P0、必做** | 软（prompt 约束）| `prompts/action-<type>.md` |
| 5. cross-action 一致性自检 | P2、V0.6.4+ | 硬 | runner |
| 6. textarea placeholder 动态 | P1 | 软 | UI dialog |

### 6.2 门槛 1：action 前置准入条件

每个 action 触发前、runner 检查前置条件、不满足拒绝。

| action | 准入条件 | 类型 |
|---|---|---|
| plan | 永远可（除非 task 已 archived）| - |
| build | 至少 1 个 plan action | **硬** |
| review | 至少 1 个 plan + 1 个 build | **硬** |
| ship | 至少 1 个 build + git status 有改动 | **硬** |
| test | 至少 1 个 build | **硬** |
| test（contextDocs 里有「测试用例」doc）| 同上 | **软 warning** |
| learn | `repoStatus = "merged"` + 之前没跑过 learn | **硬** |
| chat | 永远可 | - |

UI 实现：
- 推进 dialog 列出所有 action 选项、不满足条件的灰掉 + hover 显示原因
- runner 层 double check：用户绕过 UI（API 直调）也要拒绝

### 6.3 门槛 2：action 后置 deterministic 检查

action 跑完、runner **不靠 agent 自称「完成」**、强制跑 deterministic 检查。

| action | 后置检查 | 失败处理 |
|---|---|---|
| plan | artifact 文件存在 + 内容长度 >= 100（V0.6.0.1 起、黑名单 grep 已删） | action 标 ❌、artifact 标 needs_revise |
| build | `pnpm typecheck` exit 0 + `pnpm lint` exit 0 + git status 有改动 | action 标 ❌、提示用户「跑 build 没真改东西」|
| review | git diff hash 跟 artifact 里写的一致 + 4 类差异段非空 | action 标 ❌ |
| ship | `git push` + `gh pr create` exit 0、URL 非空 | action 标 ❌、不更新 `Task.mrs` |
| test | pass 率 ≥ 阈值（默认 100%）+ skip 率 < 50% | action 标 ❌、提示用户决策 |
| learn | propose 段有内容 + evidence 路径都能 read 到 | action 标 ❌ |
| chat | 无 | - |

**实现位置**：每个 action runner 跑完后、跑这套检查。检查脚本可以下沉到 `src/lib/server/action-checks/<type>.ts` 一份一文件、易维护。

### 6.4 门槛 3：默认 action = 推荐项（软引导）

UI 推进 dialog 的默认选中项不固定为「build」、根据 task 当前状态智能推荐：

| 当前状态 | 推荐 default |
|---|---|
| 没任何 action | plan |
| 最近一个是 plan | build |
| 最近一个是 build | review |
| 最近一个是 review、未提 MR | ship |
| 最近一个是 ship、`repoStatus = developing` | （UI 提示「等测试」或「标记完成」、不推 action）|
| `repoStatus = has_bug` | build（带 placeholder「修哪个 bug」）|
| `repoStatus = merged` | learn |
| `repoStatus = abandoned` | （不推、UI 提示「task 已关闭」）|

用户可覆盖。

### 6.5 门槛 4：action 级 anti-patterns prompt

每个 `prompts/action-<type>.md` 头部、写明该 action 的反例 / 红线（沿用 V0.5 phase-N prompt 里那些约束、按 action 重组）。

例如 `prompts/action-build.md`：

```markdown
## ⚠️ 不要做的事

1. 不读上游最新 plan / 上一个 build 就改代码
2. 改最新 plan §5 范围外的文件（防野跑）
3. 跳过 typecheck / lint（runner 兜底、但 prompt 也要明说）
4. ...（其他 V0.5 phase-2-build.md 里的约束沿用）
```

### 6.6 门槛 5：cross-action 一致性自检（V0.6.4+ 再做）

跨 action 看一致性、防 agent 在不同 action 里说话不一致。

例子：

| 检查 | 触发时机 |
|---|---|
| review 写「task 1 已完成」、对照最新 build 里 task 1 状态确实是 ✓ | review 跑完 |
| ship 写 commit 改了 N 个文件、对照 git diff 实际改的 N 个 | ship 跑完 |
| learn propose「应该用 useRequest」、对照 build artifact 里确实用了 useRequest | learn 跑完 |

实现复杂、deterministic（grep + 文件对比）、第一版 V0.6.0 不做、留到 V0.6.4+。

### 6.7 门槛 6：textarea placeholder 动态

推进 dialog 的 textarea placeholder 随选中的 action 动态变化、软引导用户写清楚指令：

| action | placeholder |
|---|---|
| plan | 「需求是什么？要解决什么问题？」（首次）/ 「方案要怎么调整？」（再次）|
| build | 「具体改什么？指向哪个文件 / 函数 / bug」|
| review | （可选）「特别关注什么？」|
| ship | （可选）「PR 标题 / 描述补充」|
| test | 「跑哪些 case？默认全跑」|
| learn | （不需要 textarea、看 propose 列表）|
| chat | 「问点啥 / 让 AI 看看 X」|

### 7.3 `_super.md` 主要改动

旧（V0.5）：

```

**{{workflowDisplayName}}**
...
{{phaseTable}}  ← plan / build / review 三行
...
关键规则 1-8
{{phasePromptSections}}  ← 三个 phase prompt 拼一起
{{startInstruction}}
```

新（V0.6）：

```

每次推进、用户在 UI 上选「下个 action 是什么」+ 写一段指令、你收到后跑对应 action。

{{actionTable}}  ← 7 行：plan/build/review/ship/test/learn/chat、每行一句话说明

1. read 用户指令（在 USER_INSTRUCTION 段）

3. 按 action-<type>.md 的指令跑
4. 写 artifact 到 actions/N-<type>.md（N 是 USER_INSTRUCTION 段告诉你的）

## 关键规则（V0.5 沿用、按 action 视角改）

（沿用 V0.5 _super.md 大部分内容、改「phase」→「action」）

{{sharedRules}}
{{actionPromptSections}}  ← 用户选了哪个 action、就 inject 哪个 action-<type>.md（不是全部拼一起）
```

**核心变化**：
- 不再前置告诉 agent「先 plan 后 build 后 review」
- 改为「用户每次告诉你下个 action 是什么、你按 action-<type>.md 跑」

## 9. 「再聊聊」/ revise / chat 在新架构下的语义

### 9.1 「再聊聊」（revise）

**语义**：针对**当前 action** 的修订、不切换 action。

行为完全沿用 V0.5.10 二分类铁则：
- 问类（纯疑问句）→ assistant_message 答疑、不动 artifact、再次 wait_for_user
- 改类（其他）→ ask_user 复述「我打算 X、对吗?」→ ✅ 后 edit 当前 action 的 artifact、再次 wait_for_user

**核心约束**：revise 后产出的 artifact 还是同一个文件（如 `actions/4-ship.md`）、**不创建新 action**。

不同 action 下 revise 的限制：
- ship action：MR 已提、revise 不能撤回外部副作用、只能修 artifact 文字或 agent 调 gh CLI 改 PR title/body
- learn action：revise 改 propose 列表、用户可重新勾选
- chat action：「再聊聊」就是继续聊、本质无差异

### 9.2 chat action

**跟 V0.5 chat 模式的关系**：吸收。

V0.5 chat-runner.ts → 删 / 合并入 plan-runner.ts（不再叫 plan-runner、可能改名 `task-runner.ts`）。

新架构下：
- 用户新建 task、可以「跳过 plan、直接 chat」（chat action 准入永远可、第 6.2 节）
- 也可以「plan 一半改用 chat 讨论」
- chat 跟其他 action 在同一个 timeline 里、不是两套并行

**为什么这么改**：
- 用户视角「chat / plan workflow」分两套是历史包袱、用户实际经常想互转
- 新架构下 chat 只是 action 类型的一种、没有特别处理

### 9.3 cancel / abandon

- **cancel 单个 action**：action 跑一半、用户想停 → 走 ack dialog「取消任务」（**未来可以加「只取消本 action 不取消 task」、V0.6.0 暂不做**）
- **abandon task**：UI 上「任务完成」改成 dropdown「已合入」/「不做了 abandon」 → 选 abandon、Run 退出、task 归档

## 10. 单 SDK Run 永生策略

### 10.1 为什么保留单 Run

- 省 Cursor 计费（沿用 V0.5）
- 保活机制 V0.3.5 shell + curl long-poll **不动**
- agent 持有上下文、跨 action 不丢

### 10.2 Run 退出时机

| 触发 | Run 行为 |
|---|---|
| action 跑完、ack approve | 不退出、调 wait_for_user 等下一 action 指令 |
| action 跑完、ack revise | 不退出、按 V0.5.10 二分类继续 |
| 用户点「任务完成（合入）」 | 触发 learn action → learn ack approve → Run 收到 `[TASK_DONE]` → 退出 |
| 用户点「任务完成（abandon）」 | Run 收到 `[TASK_ABANDONED]` → 退出 |
| Cancel chat / 用户点取消任务 | Run 收到 `[CANCELLED]` → 退出 |
| NGHTTP2 / Stream closed | Run 死掉、下次推进时自动新建 agent（沿用 V0.5 fallbackFork 思路、但不再叫 fork）|

### 10.3 wait_for_user 协议改动

V0.5：

```typescript
wait_for_user(task_id, phase, artifact)
```

V0.6：

```typescript
wait_for_user(task_id, action_id, artifact_path)
```

差异：参数从 `phase` 改为 `action_id`、artifact_path 是相对路径（之前是相对、不变）。

shell + curl long-poll 路由不变、只是查询 key 从 phase → action_id。

### 10.4 推进 → agent 联动

用户在推进 dialog 选「下个 action = build」+ 写「修取消按钮 click 没响应」、点推进：

1. fe-ai-flow 后端创建 `ActionRecord`（id = act_5、n = 5、type = build、status = running）
2. fe-ai-flow 通过 wait-ack 长连接 write 一行：`[NEXT_ACTION action_id=act_5 type=build instruction="修取消按钮 click 没响应" artifact_path=actions/5-build.md n=5]`
3. agent 的 shell + curl 拿到这行、shell exit 0
4. agent 按 super-prompt 教的、read 必要的上游 artifact、按 `action-build.md` 跑
5. 写完调 wait_for_user(task_id, act_5, "actions/5-build.md")
6. 后端跑后置检查（typecheck/lint/git status）
7. 检查通过、状态变 awaiting_ack、用户 ack
8. ack approve → wait-ack write `[ACTION_ACK approve]` → agent 拿到 → 继续 wait_for_user 等下一 action 指令

### 13.1.5 额外拍板的工作流细节（接力 AI 跟用户对齐时新增）

| 项 | 拍板结果 |
|---|---|
| **branch checkout 时机** | **build action 第一次跑前** checkout（不是 task 创建时、避免空 branch 渣沉；也不是 ship 时、避免 build 期间无 branch）。runner 检测 `task.gitBranch?.checkedOut` 为 false 时、prompt inject「先 `git checkout -b <branch> origin/<mainBranch>`、再写代码」|
| **多次 ship 是否复用 branch** | **全 task 一条 branch、复用**（V0.6-REFACTOR.md §C.4 默认方案改为「同一 branch、累计 commit」、原 MR 自动更新）。修 MR review 反馈直接接着 push、不重开 MR。|

| **Task.gitBranch** | 由 runner 在 build 第一次前生成（不是 schema 创建时填）、不可改写（防 agent 中途换 branch 混乱）|

## 附录 C：一些可能踩的坑

### C.1 super-prompt 拼接长度

V0.6 把所有 action prompt 都拼到 super-prompt 里、长度会膨胀。建议：
- 不要全拼、只 inject 当前选中 action 的 prompt（实现：`{{actionPromptSections}}` 占位符根据 `currentActionType` 注入）
- _shared.md 内容保持精简、复用 V0.5 的就 OK

### C.2 action_id 跟 artifact 文件名的耦合

artifact 文件名是 `N-<type>.md`、N 是 ActionRecord.n。如果用户在跑某个 action 时中途取消、ActionRecord 被删？还是保留 status=cancelled？

**拍板**：保留 status=cancelled、不删 ActionRecord、不释放 N（防止文件名 race）。N 单调递增。

### C.3 Run 死掉 / NGHTTP2 时怎么 resume

V0.5 fallbackFork 已经处理一部分（plan 模式）。V0.6 需要：
- Run 死时、把当前 ActionRecord status 改成 error
- 下次推进、UI 自动选「resume 死掉的 action」（不再叫 resume / fork 三选一、用户无感）
- 后端逻辑：检查 lastAgentId 是否活、活就 send 续接（相当于 resume）、死就 Agent.create 新 agent 但保留 task 上下文（相当于 fork）

### C.4 多 MR 时的 commit base

第 N 次 ship、commit base 是 main 还是上次 ship 的 branch？
- 默认：每个 ship 起独立 branch（feature/<task-id>-v1 / feature/<task-id>-v2 / ...）
- 用户拍板可改成「同一 branch、累计 commit」（但 MR 顺序冲突难处理）
